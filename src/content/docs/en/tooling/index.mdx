If you have shipped a Next.js application you already own the mental model for this module: a CLI that drives dev/build/preview, TypeScript checking, a configuration file, an integration ecosystem, and a deployment target. Astro maps one-to-one to every one of those tools — the commands and file names just change.

## The toolchain at a glance

| Next.js / React | Astro equivalent |
|---|---|
| `next dev` | `astro dev` |
| `next build` | `astro build` |
| `next start` | `astro preview` |
| `next lint` | `astro check` (type-checks `.astro` files) |
| `next.config.js` | `astro.config.mjs` |
| Next.js plugins (webpack) | Astro integrations (`@astrojs/*`) |
| `npm install @next/bundle-analyzer` | `npx astro add react` / `npx astro add tailwind` |
| Vercel / Netlify for Next.js | Vercel / Netlify adapters for Astro |
| `tsc --noEmit` | `astro check` covers `.astro` + `.ts` |
| Jest / Vitest | Vitest (same config; Astro has no test runner) |

## Lessons in this module

| Lesson | What you will learn |
|---|---|
| **Astro CLI** | `astro dev` / `build` / `preview` / `add` / `check`; `package.json` scripts |
| **Integrations** | `npx astro add`, `astro.config.mjs`, the integration ecosystem |
| **`astro check`** | Type-checking `.astro` files, editor tooling, vs `tsc`/ESLint |
| **Static vs SSR** | `output: 'static'` vs `output: 'server'`, adapters, `prerender` |
| **Deploy** | `astro build`, static hosts, SSR adapters, GitHub Actions CI |

<Diff title="Next.js toolchain vs Astro toolchain">
- Both use Vite under the hood (Astro v3+) and standard `npm`/`pnpm`/`yarn` scripts.
- `astro check` replaces the combination of `tsc --noEmit` + an ESLint rule for `.astro` syntax.
- `astro build` always outputs to `dist/` — no `.next/` cache directory to clean.
- Adapters (Vercel, Netlify, Node) replace the concept of Next.js deployment targets.
</Diff>
